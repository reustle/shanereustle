Running MongoDB on Amazon Web Services

3 March 2012

I've spent the past few months managing and scaling multiple MongoDB clusters on AWS, and have picked up a few tricks along the way. Below I've explained a few of the odd or helpful things I've learned from setting up the instance to maintaining it.

Whenever I'm starting an instance from scratch, rather than one of my own AMIs, I will use an Ubuntu AMI from Eric Hammond. They make it really easy to get an instance up and running quickly. One thing to note with picking which AMI to use is that Ubuntu 10.04 does not play well with EBS. I had all kinds of weird problems including Mongo locking up entirely for 5 minutes at a time due to 100% disk utilization. After getting everything over to 11.10, Mongo was fine. If you find yourself installing a lot of the same software over and over after setting up an instance, you may want to consider making your own AMI on top of one of these.

A good practice when setting up a fresh instance for Mongo is to raise the limit to the number of open files allowed. You can also do this after you have a machine already running, but the change requires a restart. To raise the limit, you'll need to modify /etc/sysctl.conf and /etc/security/limits.conf. The changes that need to be made are listed below.

Overwrite /etc/sysctl.conf with 
fs.file-max = 360000
net.ipv4.ip_local_port_range = 1024 65000
net.ipv4.tcp_max_syn_backlog = 2048
Overwrite /etc/security/limits.conf with 
ubuntu     soft     nofile     350000
ubuntu     hard     nofile     350000

If you're looking to get reasonably good performance out of EBS, you're going to want to set up a RAID drive with 4x 1TB drives. In a blog post on aws drive performance, Victor Trac did a few tests on how to get the most performance out of EBS, and he found that this was the best case. If you're going to want to set this up, you can use the shell script I have below. I don't know the original author, but I modified it to work with Ubuntu 11.04 and above, since they went from using sdX as their drive names to xvdX. what is setra and what should it be, same with chunk

raid.sh 
# mdadm == Raiding software
# xfsprogs == XFS filesystem creator/manager

sudo apt-get install -y mdadm xfsprogs;

sudo mdadm --create /dev/md0 --level 0 --chunk=256 --metadata=1.1 --raid-devices=4 /dev/xvdf /dev/xvdg /dev/xvdh /dev/xvdi;
echo DEVICE /dev/xvdf /dev/xvdg /dev/xvdh /dev/xvdi  | sudo tee /etc/mdadm/mdadm.conf;
sudo mdadm --detail --scan | sudo tee -a /etc/mdadm/mdadm.conf;

sudo mkfs.xfs /dev/md0;
echo "/dev/md0 /raiddrive xfs noatime 0 0" | sudo tee -a /etc/fstab;

sudo mkdir /raiddrive;
sudo mount /raiddrive;

sudo blockdev --setra 65536 /dev/md0;

df -h /raiddrive;

sudo chown ubuntu:ubuntu /raiddrive;

When you get around to starting Mongo, you're going to need to make some decisions on things like where the log file is located (--logpath). I have found that putting the log file on a separate disk from the actual DB is a good idea. The log file can sometimes get unruly and can be stealing valuable resources from your primary DB disk.

When running Mongo on AWS, there are a few things you need to take into account. One of the bigger factors is how much data do you need to store, and how fast do you need it. According to this blog post by foo bar, you will get the most performance out of AWS drives by using raid-0 with 4 EBS drives. It is also wise to use EBS because if your data is on ephemeral storage and the machine restarts for whatever reason, your data is gone. You can snapshot EBS. Here is raid.sh and what it does.

When starting mongo, these are the flags I use.

  • --fork : Send itself to a backgrounded process
  • --logpath : Where to store the log file (different disk)
  • --dbpath : Where to store the actual db files, your raiddrive if you set one up
  • --nojournal : don't journal, because x
  • --rest : rest interface
  • whatever else you want (replica set stuff, ports, etc)
